## Knowing-Bot

보안과 권한 통제를 우선으로 설계한 사내 지식 AI + 외부 상담 AI 분리형 업무 플랫폼 프로젝트.
단순히 “답변을 잘하는 챗봇”이 아니라, 누가 어떤 문서를 볼 수 있는지, 근거 없는 답변을 어떻게 막을지, 감사로그와 권한 승인 흐름을 어떻게 남길지까지 포함해 실제 기업 도입을 염두에 두고 설계한 프로젝트입니다.

## 프로젝트 소개

많은 조직이 사내 문서를 갖고 있어도, 실제 현장에서는
문서를 찾기 어렵고, 최신 버전이 무엇인지 헷갈리며,
권한이 없는 정보가 섞이거나 외부 상담에 내부 정보가 노출될 위험이 있습니다.

Knowing-Bot은 이 문제를 해결하기 위해,
내부 직원용 지식 AI와 외부 고객용 상담 AI를 구조적으로 분리하고,
내부 AI에는 출처 강제형 응답, ACL 기반 문서 필터링, 임시 권한 요청/승인, 감사로그를 결합한 보안 우선형 구조를 제안합니다.

## 왜 이 프로젝트를 만들었는가

저는 AI 기능 자체보다, 실제 서비스로 배포했을 때 생기는 보안과 운영 문제가 더 중요하다고 생각했습니다.
그래서 이 프로젝트에서는 “AI를 붙였다”에서 끝내지 않고,
다음 질문에 답할 수 있는 구조를 만드는 데 집중했습니다.

- 근거 없는 답변을 어떻게 줄일 것인가
- 외부 상담 AI가 내부 문서에 절대 닿지 않게 하려면 어떻게 분리할 것인가
- 권한 없는 문서 요청이 들어왔을 때 어떤 UX와 승인 흐름이 필요한가
- 도입 기업이 감사·로그·배포·권한 회수까지 설명 가능한 구조는 무엇인가
## 핵심 설계 포인트
- 1. 내부 AI / 외부 AI 이중 분리
내부 직원용 AI는 로그인 후 사용하며, 역할과 권한 범위 내에서만 문서를 검색합니다.
외부 고객용 AI는 FAQ/승인된 상담 시나리오만 사용하며, 내부 문서에는 접근하지 않습니다.
- 2. 출처 강제형 응답
내부 AI는 ACL을 통과한 문서만 검색하고,
근거가 충분할 때만 답변하며,
답변 하단에 출처를 표시하고 원문 위치로 이동할 수 있도록 설계했습니다.
- 3. 권한 요청과 만료까지 포함한 접근 제어
권한이 없는 자료는 결과 단계에서 제거하고,
사용자는 필요한 문서/범위에 대해 사유를 입력해 권한을 요청할 수 있으며,
승인 시 TTL이 있는 임시 권한을 발급하고 만료 시 자동 회수되도록 설계했습니다.
- 4. 감사 가능한 AI 시스템
누가 언제 질문했는지,
어떤 문서가 검색됐는지,
어떤 출처가 응답에 포함됐는지,
어떤 권한 요청과 승인 이력이 있었는지까지 추적 가능한 운영 구조를 목표로 했습니다.
## 기술 스택

Frontend
Next.js, TypeScript, shadcn/ui, Tailwind CSS

Backend
Node.js, TypeScript, Next.js API 또는 별도 API 서버, 권한 판정 서비스, 검색/인덱싱 서비스, 상담 플로우 서비스

Data / Infra
PostgreSQL, Redis(선택), AWS ECS Fargate, RDS PostgreSQL, S3, CloudFront, Route 53, WAF, Secrets Manager, KMS

개발 관점에서 강조하고 싶은 점

이 프로젝트에서 제가 가장 중요하게 본 것은 “챗봇 기능 추가”가 아니라,
기업 환경에서 AI를 안전하게 쓸 수 있게 만드는 설계였습니다.

특히 아래 역량을 보여주기 위한 프로젝트로 만들었습니다.

- 보안 요구사항을 제품 구조로 번역하는 능력
- 권한/감사/출처/배포까지 고려한 백엔드 중심 사고
- 단순 CRUD를 넘어 실제 운영 시나리오를 설계하는 능력
- AI 기능을 제품 기능이 아니라 서비스 시스템으로 바라보는 시각
## 현재 레포 상태
